# 5.10 Identifying Endpoints

Before fuzzing any Web API, you must first identify its **endpoints**, which are the URLs where the API accepts requests. These endpoints act like entry points to access or manipulate data.

***

### **REST API**

In **REST APIs**, endpoints are structured URLs representing resources. For example, `/users` returns all users, while `/users/123` returns a specific user. REST APIs also use parameters to control behavior. Query parameters are added after `?` for filtering or pagination like `/users?limit=10`, path parameters are part of the URL like `/products/123`, and request body parameters are sent in POST or PUT requests to create or update data.

Parameters are used to modify the behavior of API requests or provide additional information. In REST APIs, there are several types of parameters:

| Parameter Type          | Description                                                                                           | Example                                     |
| ----------------------- | ----------------------------------------------------------------------------------------------------- | ------------------------------------------- |
| Query Parameters        | Appended to the endpoint URL after a question mark (`?`). Used for filtering, sorting, or pagination. | `/users?limit=10&sort=name`                 |
| Path Parameters         | Embedded directly within the endpoint URL. Used to identify specific resources.                       | `/products/{id}pen_spark`                   |
| Request Body Parameters | Sent in the body of POST, PUT, or PATCH requests. Used to create or update resources.                 | `{ "name": "New Product", "price": 99.99 }` |

#### Discovering REST Endpoints and Parameters <a href="#discovering-rest-endpoints-and-parameters" id="discovering-rest-endpoints-and-parameters"></a>

To discover REST endpoints, the easiest method is checking API documentation like Swagger or OpenAPI. If not available, you can analyze network traffic using tools like Burp Suite or browser developer tools to observe requests. You can also use fuzzing tools like ffuf to find hidden parameters:

```bash
ffuf -u http://example.com/api/users?FUZZ=test -w wordlist.txt
```

This command tries different parameter names from the wordlist to discover hidden parameters.

***

### SOAP API <a href="#soap" id="soap"></a>

In **SOAP APIs**, there is usually a single endpoint (like `/soap`) and operations are defined inside XML messages. The structure and available operations are described in a **WSDL file**, which is the most important source for understanding SOAP APIs. A SOAP request looks like:

```xml
<soapenv:Envelope>
  <soapenv:Body>
    <SearchBooks>
      <keywords>cybersecurity</keywords>
      <author>Dan Kaminsky</author>
    </SearchBooks>
  </soapenv:Body>
</soapenv:Envelope>
```

Here, parameters are inside XML tags. You can discover SOAP operations by analyzing the WSDL file or capturing traffic using tools like Wireshark or Burp Suite. Fuzzing can also be done by modifying XML values.

***

### Identifying GraphQL API Endpoints and Parameters <a href="#identifying-graphql-api-endpoints-and-parameters" id="identifying-graphql-api-endpoints-and-parameters"></a>

In **GraphQL APIs**, there is usually a single endpoint like `/graphql`. Instead of multiple URLs, GraphQL uses queries and mutations to fetch or modify data.

#### GraphQL Queries <a href="#graphql-queries" id="graphql-queries"></a>

Queries are designed to fetch data from the GraphQL server. They pinpoint the exact fields, relationships, and nested objects the client desires, eliminating the issue of over-fetching or under-fetching data common in REST APIs. Arguments within queries allow for further refinement, such as filtering or pagination.

| Component     | Description                                                                               | Example                                                   |
| ------------- | ----------------------------------------------------------------------------------------- | --------------------------------------------------------- |
| Field         | Represents a specific piece of data you want to retrieve (e.g., name, email).             | `name`, `email`                                           |
| Relationship  | Indicates a connection between different types of data (e.g., a user's posts).            | `posts`                                                   |
| Nested Object | A field that returns another object, allowing you to traverse deeper into the data graph. | `posts { title, body }`                                   |
| Argument      | Modifies the behavior of a query or field (e.g., filtering, sorting, pagination).         | `posts(limit: 5)` (retrieves the first 5 posts of a user) |

```graphql
query {
  user(id: 123) {
    name
    email
  }
}
```

In this example:

* We query for information about a `user` with the ID 123.
* We request their `name` and `email`.
* We also fetch their first 5 `posts`, including the `title` and `body` of each post.

#### GraphQL Mutations <a href="#graphql-mutations" id="graphql-mutations"></a>

Mutations are the counterparts to queries designed to modify data on the server. They encompass operations to create, update, or delete data. Like queries, mutations can also accept arguments to define the input values for these operations.

| Component | Description                                                                                             | Example                                                          |
| --------- | ------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| Operation | The action to perform (e.g., createPost, updateUser, deleteComment).                                    | `createPost`                                                     |
| Argument  | Input data required for the operation (e.g., title and body for a new post).                            | `title: "New Post", body: "This is the content of the new post"` |
| Selection | Fields you want to retrieve in the response after the mutation completes (e.g., id, title of new post). | `id`, `title`                                                    |

```graphql
mutation {
  createPost(title: "Test", body: "Hello") {
    id
    title
  }
}
```

This mutation creates a new post with the specified title and body, returning the `id` and `title` of the newly created post in the response.

To discover GraphQL endpoints and parameters, you can use **introspection queries** to get the full schema, which shows all available queries and mutations. You can also analyze traffic or use tools like GraphiQL. Fuzzing can be applied to arguments and fields to test for weaknesses.
