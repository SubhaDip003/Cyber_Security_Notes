# 7.1 Introduction

Most modern web applications use a database on the back-end. A database is used to store and manage important data such as website content, user accounts, passwords, messages, and other information.

To make a website dynamic and interactive, the web application must communicate with the database in real time. When a user sends an HTTP/HTTPS request, the web server processes the request and sends a query to the database to get or update data. The result is then shown to the user.

A common structure used in web applications is the **Three-Tier Architecture**:

1. **Tier 1 – Client/User Interface:** The user interacts with the website or application through a browser.
2. **Tier 2 – Application Server:** The server handles user requests and communicates with the database.
3. **Tier 3 – Database Server (DBMS):** Stores and manages the application data.

If user input is directly included in database queries without proper validation or security checks, an attacker can manipulate the query. This can cause the database to perform unintended actions. This type of attack is called **SQL Injection (SQLi)**.

**SQL Injection** is an attack used against relational databases such as MySQL, where attackers inject malicious SQL commands into input fields to access, modify, or delete data.

Attacks against non-relational databases such as MongoDB are known as **NoSQL Injection**.

This topic mainly focuses on SQL Injection using MySQL to understand the basic concepts.

***

### SQL Injection (SQLi)

Many types of injection attacks can happen in web applications, such as HTTP injection, code injection, and command injection. The most common one is **SQL Injection (SQLi)**.

SQL Injection happens when an attacker enters malicious input into a website form or URL, which changes the SQL query sent by the web application to the database. This allows the attacker to run unauthorized database queries.

There are different ways to perform SQL Injection. First, the attacker tries to break the normal input by using special characters like single quote **(')** or double quote **(")**. This helps them escape the user input field and insert SQL code into the query.

After that, the attacker modifies the original query or creates a new query to get unauthorized access to data. This can be done using techniques such as:

* **Stacked Queries** – Running multiple SQL queries together.
* **UNION Queries** – Combining results from another query.

Finally, the attacker views or captures the returned data through the web application's front end.

SQL Injection is dangerous because it can allow attackers to read, modify, or delete sensitive database information.

***

### Use Cases and Impact of SQL Injection

SQL Injection can cause serious damage, especially when the server and database have weak security settings.

One major impact is stealing sensitive information that should stay private. Attackers may access usernames, passwords, personal details, or credit card data. This stolen data can be used to hack accounts, commit fraud, or misuse other online services.

Another common use is bypassing the normal login system. An attacker may log in without using a correct username and password. They may also access restricted areas of the website, such as admin panels or private user sections.

In some cases, attackers can read, change, or delete data stored in the database. They may also upload harmful files or create hidden access methods on the server. This can allow them to take control of the website and damage the whole system.

***

### Prevention <a href="#prevention" id="prevention"></a>

SQL injections are usually caused by poorly coded web applications or poorly secured back-end server and databases privileges. Later on, we will discuss ways to reduce the chances of being vulnerable to SQL injections through secure coding methods like user input sanitization and validation and proper back-end user privileges and control.
