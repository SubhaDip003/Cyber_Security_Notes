# 7.9 Subverting Query Logic

Now that we have a basic idea about how SQL statements work let us get started with SQL injection. Before we start executing entire SQL queries, we will first learn to modify the original query by injecting the `OR` operator and using SQL comments to subvert the original query's logic. A basic example of this is bypassing web authentication, which we will demonstrate in this section.

***

### Authentication Bypass <a href="#authentication-bypass" id="authentication-bypass"></a>

Consider the following administrator login page.

<figure><img src="../.gitbook/assets/image (123).png" alt=""><figcaption></figcaption></figure>

We can log in with the administrator credentials <mark style="color:$info;">`admin`</mark>` ``/`` `<mark style="color:$info;">`p@ssw0rd`</mark>.

<figure><img src="../.gitbook/assets/image (124).png" alt=""><figcaption></figcaption></figure>

The page also displays the SQL query being executed to understand better how we will subvert the query logic. Our goal is to log in as the admin user without using the existing password. As we can see, the current SQL query being executed is:

```sql
SELECT * FROM logins WHERE username='admin' AND password = 'p@ssw0rd';
```

The page takes in the credentials, then uses the <mark style="color:$info;">`AND`</mark> operator to select records matching the given username and password. If the <mark style="color:$info;">`MySQL`</mark> database returns matched records, the credentials are valid, so the <mark style="color:$info;">`PHP`</mark> code would evaluate the login attempt condition as <mark style="color:$info;">`true`</mark>. If the condition evaluates to <mark style="color:$info;">`true`</mark>, the admin record is returned, and our login is validated. Let us see what happens when we enter incorrect credentials.

<figure><img src="../.gitbook/assets/image (125).png" alt=""><figcaption></figcaption></figure>

As expected, the login failed due to the wrong password leading to a <mark style="color:$info;">`false`</mark> result from the <mark style="color:$info;">`AND`</mark> operation.

### SQLi Discovery

Before trying to bypass a login page or change application logic, we first need to check whether the login form is vulnerable to **SQL Injection**.

To test this, special characters are entered in the username field. If the page shows an error or behaves differently, it may indicate SQL Injection vulnerability.

Some common test payloads are:

| Payload | URL Encoded |
| ------- | ----------- |
| `'`     | `%27`       |
| `"`     | `%22`       |
| `#`     | `%23`       |
| `;`     | `%3B`       |
| `)`     | `%29`       |

Sometimes the URL encoded version is needed, especially when the payload is placed directly in the URL using an HTTP GET request.

For example, if we enter a single quote `'` in the username field and submit the login form, the application may return an SQL error instead of a normal **Login Failed** message.

<figure><img src="../.gitbook/assets/image (2) (1).png" alt=""><figcaption></figcaption></figure>

The back-end query may look like:

```sql
SELECT * FROM logins WHERE username='''AND password='something';
```

The application originally expected something like:

```sql
SELECT * FROM logins WHERE username='admin'AND password='something';
```

But after adding the single quote, the total number of quotes becomes incorrect. This creates an invalid SQL query and causes a syntax error.

When a website shows SQL errors after entering special characters, it is often a strong sign that user input is being placed directly into the query without proper sanitization.

This type of testing is called **SQLi discovery**, where attackers or security testers try small payloads to identify vulnerable input fields.

***

### OR Injection

**OR Injection** is a common SQL Injection technique used to bypass login authentication. The goal is to make the SQL query return <mark style="color:$info;">`TRUE`</mark>, even when the password is incorrect.

Many login forms use a query like this:

```sql
SELECT * FROM logins WHERE username='admin' AND password='pass123';
```

This query only returns a result when both the username and password are correct.

If user input is not sanitized, an attacker may inject the <mark style="color:$info;">`OR`</mark> operator into the username field.

Example payload:

```sql
admin' OR '1'='1
```

If this value is entered in the username field, the final query may become:

```sql
SELECT * FROM logins WHERE username='admin' OR '1'='1' AND password='something';
```

Now let us understand how this works.

The condition:

```sql
'1'='1'
```

is always <mark style="color:$info;">**`TRUE`**</mark> because <mark style="color:$info;">1 is equal to 1</mark>.

In MySQL, the <mark style="color:$info;">`AND`</mark> operator is checked before <mark style="color:$info;">`OR`</mark>. As previously discussed, the MySQL documentation for [operation precedence](https://dev.mysql.com/doc/refman/8.0/en/operator-precedence.html) states that the <mark style="color:$info;">`AND`</mark> operator would be evaluated before the <mark style="color:$info;">`OR`</mark> operator.

So the database first checks:

```sql
'1'='1' AND password='something'
```

If the password is wrong, this part becomes <mark style="color:$info;">`FALSE`</mark>.

Then the database checks:

```sql
username='admin' OR FALSE
```

If the username **admin** exists in the table, the first condition becomes <mark style="color:$info;">`TRUE`</mark>.

So:

```sql
TRUE OR FALSE = TRUE
```

<figure><img src="../.gitbook/assets/image (3) (1).png" alt=""><figcaption></figcaption></figure>

Let us break it down:

The <mark style="color:$info;">`AND`</mark> operator is evaluated first:

* <mark style="color:$info;">`'1'='1'`</mark> is <mark style="color:$info;">`True`</mark>.
* <mark style="color:$info;">`password='something'`</mark> is <mark style="color:$info;">`False`</mark>.
* The result of the <mark style="color:$info;">`AND`</mark> condition is <mark style="color:$info;">`False`</mark> because <mark style="color:$info;">`True AND False`</mark> is <mark style="color:$info;">`False`</mark>.

Next, the <mark style="color:$info;">`OR`</mark> operator is evaluated:

* If <mark style="color:$info;">`username='admin'`</mark> exists, the entire query returns <mark style="color:$info;">`True`</mark>.
* The <mark style="color:$info;">`'1'='1'`</mark> condition is irrelevant in this context because it doesn't affect the outcome of the <mark style="color:$info;">`AND`</mark> condition.

Therefore, the query will return <mark style="color:$info;">`True`</mark> if a username <mark style="color:$info;">`'admin'`</mark> exists, bypassing authentication.

Because one side of the <mark style="color:$info;">`OR`</mark> condition is <mark style="color:$info;">`TRUE`</mark>, the whole query becomes <mark style="color:$info;">`TRUE`</mark>, and the attacker may log in without knowing the password.

This means the application may allow access to the **admin** account simply because the username exists.

OR Injection is possible only when user input is directly inserted into SQL queries without protection.

> Note: The payload we used above is one of many auth bypass payloads we can use to subvert the authentication logic. You can find a comprehensive list of SQLi auth bypass payloads in [PayloadAllTheThings](https://github.com/swisskyrepo/PayloadsAllTheThings/tree/master/SQL%20Injection#authentication-bypass), each of which works on a certain type of SQL queries.
