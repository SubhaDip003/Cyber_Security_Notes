# 5.8 Validating Findings

Fuzzing helps discover many possible issues, but not all results are real vulnerabilities. Some findings may be false positives, so validation is necessary to confirm whether the issue is genuine and to understand its real impact. Validation also helps reproduce the issue and collect proper evidence for reporting.

The most reliable way to validate a finding is through **manual verification**. This means repeating the same request that triggered the result and carefully analyzing the response. Tools like curl or a browser are commonly used for this purpose. If needed, a safe proof of concept (PoC) can be created to demonstrate the issue without causing damage.

For example, if fuzzing discovers a directory like `/backup/`, it may indicate sensitive data exposure. To verify this, use:

```bash
curl http://IP:PORT/backup/
```

This command sends a request to the server and displays the response. If the output shows a directory listing (files like `backup.sql` or `password.txt`), it confirms that the directory is accessible and potentially vulnerable. This could look something like this:

To avoid accessing sensitive data directly, it is safer to check only the response headers using:

```bash
curl -I http://IP:PORT/backup/password.txt

HTTP/1.1 200 OK
Content-Type: text/plain;charset=utf-8
ETag: "3406387762"
Last-Modified: Wed, 12 Jun 2024 14:08:46 GMT
Content-Length: 171
Accept-Ranges: bytes
Date: Wed, 12 Jun 2024 14:08:59 GMT
Server: lighttpd/1.4.76
```

The `-I` option fetches only the headers, not the file content. From the response:

* **Content-Type** shows the file type (e.g., text, SQL, zip).
* **Content-Length** shows the file size. A value greater than zero indicates the file contains data.

If a file like `password.txt` exists with non-zero size inside a publicly accessible backup directory, it strongly suggests a security risk.
