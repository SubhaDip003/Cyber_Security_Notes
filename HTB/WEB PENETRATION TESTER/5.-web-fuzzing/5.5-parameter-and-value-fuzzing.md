# 5.5 Parameter and Value Fuzzing

Parameter and value fuzzing is a technique used to test how a web application handles input by **changing parameter values in requests**. Parameters act like variables that control how the application behaves.

There are two main types of parameters:

**GET Parameters** appear in the URL and are visible. They are mainly used for actions like searching or filtering. You'll often spot `GET` parameters right in the URL, following a question mark (`?`). Multiple parameters are strung together using ampersands (`&`). For example:

```http
https://example.com/search?query=fuzzing&category=security
```

**POST Parameters** are sent in the request body and are not visible in the URL. They are used for sensitive data like login forms. The data is usually encoded as `application/x-www-form-urlencoded` or `multipart/form-data`.&#x20;

When you submit a form or interact with a web page that uses POST requests, the following happens:

1. `Data Collection`: The information entered into the form fields is gathered and prepared for transmission.
2. `Encoding`: This data is encoded into a specific format, typically `application/x-www-form-urlencoded` or `multipart/form-data`:
   * `application/x-www-form-urlencoded`: This format encodes the data as key-value pairs separated by ampersands (`&`), similar to GET parameters but placed within the request body instead of the URL.
   * `multipart/form-data`: This format is used when submitting files along with other data. It divides the request body into multiple parts, each containing a specific piece of data or a file.
3. `HTTP Request`: The encoded data is placed within the body of an HTTP POST request and sent to the web server.
4. `Server-Side Processing`: The server receives the POST request, decodes the data, and processes it according to the application's logic.

Here's a simplified example of how a POST request might look when submitting a login form:

```http
POST /login HTTP/1.1
Host: example.com
Content-Type: application/x-www-form-urlencoded

username=your_username&password=your_password
```

* `POST`: Indicates the HTTP method (POST).
* `/login`: Specifies the URL path where the form data is sent.
* `Content-Type`: Specifies how the data in the request body is encoded (`application/x-www-form-urlencoded` in this case).
* `Request Body`: Contains the encoded form data as key-value pairs (`username` and `password`).

***

## Why Parameters Matter for Fuzzing <a href="#why-parameters-matter-for-fuzzing" id="why-parameters-matter-for-fuzzing"></a>

Fuzzing works by sending many different values to these parameters to see how the application responds. This helps in finding:

* Hidden features
* Incorrect validations
* Vulnerabilities like XSS or SQL Injection

For example, changing a parameter like `id=1` to another value might give access to unauthorized data.

Instead of testing manually, tools like **wenum** (for GET) and **ffuf** (for POST) automate this process by sending large numbers of payloads using wordlists.

The goal is to find a value that produces a **different or valid response (e.g., HTTP 200)**, which may indicate a vulnerability or hidden functionality.

***

## Fuzzing GET parameter and value using wenum <a href="#wenum" id="wenum"></a>

In this task, we use **wenum** to fuzz GET/POST parameters and discover valid values that trigger different server responses.

First, install wenum:

```bash
pipx install git+https://github.com/WebFuzzForge/wenum
pipx runpip wenum install setuptools
```

Then manually test the endpoint:

```bash
curl http://IP:PORT/get.php

## Invalid parameter value
## x:
```

Response shows parameter **x is missing**.

Add a value:

```bash
curl http://IP:PORT/get.php?x=1

## Invalid parameter value
## x: 1
```

Now the server recognizes `x`, but the value is invalid. This means the application checks parameter values.

Instead of guessing manually, use **wenum** to fuzz values:

```bash
wenum -w /usr/share/seclists/Discovery/Web-Content/common.txt --hc 404 -u "http://IP:PORT/get.php?x=FUZZ"
```

* `-w` → wordlist
* `--hc 404` → hide 404 responses
* `FUZZ` → replaced with wordlist values

From results, most responses are invalid, but one request returns **HTTP 200**, indicating a valid value.

Accessing that value:

```bash
curl http://IP:PORT/get.php?x=VALID_VALUE
```

This reveals the result.

## Fuzzing POST parameter and value using FFUF <a href="#wenum" id="wenum"></a>

Fuzzing POST parameters is similar to GET fuzzing, but instead of sending data in the URL, the payload is sent in the **request body**.

First, using curl with an empty POST request shows that the parameter `y` is required:

```bash
curl -d "" http://IP:PORT/post.php

## Invalid parameter value
## y:
```

The response indicates that the parameter is missing.

To automate testing different values, **FFUF** is used. The key difference is the use of the `-d` option to send POST data:

```bash
ffuf -u http://IP:PORT/post.php -X POST -H "Content-Type: application/x-www-form-urlencoded" -d "y=FUZZ" -w /usr/share/seclists/Discovery/Web-Content/common.txt -mc 200
```

Here:

* `-X POST` → specifies POST request
* `-d "y=FUZZ"` → sends fuzzing payload in request body
* `-w` → wordlist for testing values
* `-mc 200` → filters valid responses

Most responses will be invalid, but the **correct value** will return a **200 OK response** and stand out.

After finding the valid value, confirm it using curl:

```bash
curl -d "y=VALUE" http://IP:PORT/post.php
```

If correct, it will return the expected result.
