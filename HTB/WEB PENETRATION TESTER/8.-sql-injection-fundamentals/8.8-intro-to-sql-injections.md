# 8.8 Intro to SQL Injections

Now that we have a general idea of how MySQL and SQL queries work let us learn about SQL injections.

***

### Use of SQL in Web Applications

Web applications use databases like MySQL to store, manage, and retrieve data. After the database server is installed and running on the back-end, the web application connects to it and performs operations using SQL queries.

For example, in a PHP-based web application, we can connect to the database using credentials such as host, username, password, and database name. After establishing the connection, we can execute SQL queries directly from the application.

```php
$conn = new mysqli("localhost", "root", "password", "users");
$query = "SELECT * FROM logins";
$result = $conn->query($query);
```

In this example, the application connects to the database named **users** and runs a query to fetch all records from the **logins** table. The result of this query is stored in the variable `$result`.

Once the data is retrieved, the application can display it on the webpage or use it for other processing. The following PHP code shows how the fetched data can be printed line by line:

```php
while($row = $result->fetch_assoc()){
    echo $row["name"] . "<br>";
}
```

Here, each row from the database is accessed as an associative array, and the value of the **name** column is printed on the screen.

In real-world web applications, user input is often used to interact with the database. For example, when a user searches for a username, the input provided by the user is sent to the server and used inside an SQL query.

```php
$searchInput = $_POST['findUser'];
$query = "SELECT * FROM logins WHERE username LIKE '%$searchInput%'";
$result = $conn->query($query);
```

In this case, whatever the user types in the search field is stored in `$searchInput` and then used in the SQL query to find matching usernames. The `%` symbol is used as a wildcard to match any characters before or after the input.

This method allows dynamic interaction with the database, but it can be dangerous if the input is not handled properly. If user input is directly added to SQL queries without validation or filtering, it can lead to serious security issues such as SQL Injection.

SQL Injection occurs when an attacker provides malicious input that changes the structure of the SQL query. This can allow unauthorized access to data, bypass authentication, or even modify or delete database content.

Therefore, while using SQL in web applications is powerful and necessary, it is very important to handle user input securely to prevent vulnerabilities.

***

### What is an Injection?

Injection is a security vulnerability that happens when a web application accepts user input and uses it directly without proper protection.

In many applications, user input is added into commands, queries, or scripts. If the input is not checked or cleaned properly, the application may treat the user input as actual code instead of normal text.

**Sanitization** means removing or filtering special characters from user input to stop malicious data from being executed. Common special characters may include quotes, semicolons, brackets, or other symbols used in programming and database queries.

For example, if an application takes input and directly places it inside an SQL query, an attacker may insert special characters such as (<mark style="color:$info;">`'`</mark>) to break the original query structure and add their own commands.

Instead of reading the input as a normal string, the application misinterprets it as part of the code. This changes the normal flow of the program and may execute unintended commands.

Injection can happen in different technologies, such as:

* SQL Injection – malicious SQL code is inserted into database queries
* Command Injection – system commands are inserted and executed
* Cross-Site Scripting (XSS) – malicious JavaScript is injected into webpages

If user input is not sanitized or securely handled, the injected code may run successfully and cause data theft, unauthorized access, or system compromise.

Therefore, all user input should be validated, sanitized, and handled securely before being used by the application.

***

### SQL Injection

SQL Injection is a vulnerability that happens when user input is inserted into an SQL query without proper sanitization or filtering. Because of this, an attacker may change the original query and run unwanted SQL commands.

Consider the following PHP example:

```php
$searchInput = $_POST['findUser'];
$query = "SELECT * FROM logins WHERE username LIKE '%$searchInput'";
$result = $conn->query($query);
```

In this script, the value entered by the user is directly placed inside the SQL query. The application expects normal search text, but since there is no security filtering, malicious input may also be accepted.

The original SQL query format is:

```sql
SELECT * FROM logins WHERE username LIKE '%$searchInput'
```

If the user enters:

```
admin
```

Then the query becomes:

```sql
SELECT * FROM logins WHERE username LIKE '%admin'
```

This is a normal search query, and the database will search for usernames related to **admin**.

If the user enters:

```
SHOW DATABASES;
```

Then the query becomes:

```sql
SELECT * FROM logins WHERE username LIKE '%SHOW DATABASES;'
```

Here, the text is still treated as normal search data, not as SQL code.

However, if the attacker uses a single quote <mark style="color:$info;">`'`</mark>, they may break out of the string boundary and inject their own SQL commands.

Example malicious input:

```
1'; DROP TABLE users;
```

Then the input inside the query becomes:

```
'%1'; DROP TABLE users;'
```

The final query may look like:

```sql
SELECT * FROM logins WHERE username LIKE '%1'; DROP TABLE users;'
```

In this case, the first quote closes the original string, and the attacker adds a new SQL command. The command <mark style="color:$info;">`DROP TABLE users;`</mark> attempts to delete the **users** table from the database.

This shows how dangerous SQL Injection can be when applications directly trust user input.

> Note: In the above example, for the sake of simplicity, we added another SQL query after a semi-colon (;). Though this is actually not possible with MySQL, it is possible with MSSQL and PostgreSQL. In the coming sections, we'll discuss the real methods of injecting SQL queries in MySQL.

In real-world databases like MySQL, multiple queries in one statement are often restricted by default. However, attackers use many other techniques to exploit SQL Injection.

To prevent SQL Injection, developers should use input validation, sensitization, and prepared statements instead of directly placing user input into SQL queries.

***

### Syntax Errors <a href="#syntax-errors" id="syntax-errors"></a>

The previous example of SQL injection would return an error:

```php
Error: near line 1: near "'": syntax error
```

This is because of the last trailing character, where we have a single extra quote (`'`) that is not closed, which causes a SQL syntax error when executed:

```sql
select * from logins where username like '%1'; DROP TABLE users;'
```

In this case, we had only one trailing character, as our input from the search query was near the end of the SQL query. However, the user input usually goes in the middle of the SQL query, and the rest of the original SQL query comes after it.

To have a successful injection, we must ensure that the newly modified SQL query is still valid and does not have any syntax errors after our injection. In most cases, we would not have access to the source code to find the original SQL query and develop a proper SQL injection to make a valid SQL query. So, how would we be able to inject into the SQL query then successfully?

One answer is by using <mark style="color:$info;">`comments`</mark>, and we will discuss this in a later section. Another is to make the query syntax work by passing in multiple single quotes, as we will discuss next (<mark style="color:$info;">`'`</mark>).

Now that we understand SQL injections' basics let us start learning some practical uses.

***

## Types of SQL Injections <a href="#types-of-sql-injections" id="types-of-sql-injections"></a>

SQL Injections are categorized based on how and where we retrieve their output.

<figure><img src="../.gitbook/assets/image (164).png" alt=""><figcaption></figcaption></figure>

SQL Injection attacks are divided into different types based on how the attacker gets the result or output from the database.

The main categories of SQL Injection are **In-Band SQL Injection**, **Blind SQL Injection**, and **Out-of-Band SQL Injection**.

### In-Band SQL Injection

In this type, the attacker receives the result directly on the webpage. The same communication channel is used to send the attack and receive the response. This is the easiest and most common type of SQL Injection.

There are two common types of In-Band SQL Injection.

#### Union-Based SQL Injection

This method uses the SQL <mark style="color:$info;">`UNION`</mark> operator to combine the result of the original query with another query created by the attacker.

The attacker usually needs to know the correct number of columns and where the data can be displayed.

Example:

```sql
SELECT name, email FROM users
UNION SELECT database(), user();
```

This may display database name or current user details on the page.

#### Error-Based SQL Injection

This method works when the website shows database or PHP error messages. The attacker intentionally creates an error to reveal useful information.

Example:

```sql
SELECT * FROM users WHERE id='1'
```

If modified incorrectly, the database may return an error message containing table names, column names, or query details.

### Blind SQL Injection

In some websites, query results are not shown directly. The attacker must collect information indirectly by observing application behavior.

There are two common types of Blind SQL Injection.

#### Boolean-Based SQL Injection

This method uses <mark style="color:$info;">True</mark> or <mark style="color:$info;">False</mark> conditions. If the condition is true, the page behaves normally. If false, the page response changes.

Example:

```sql
1' AND 1=1 -- 
```

This may return a normal page.

```sql
1' AND 1=2 -- 
```

This may return an empty page or different response.

By checking responses, attackers can guess data character by character.

#### Time-Based SQL Injection

This method uses time delay functions. If the condition is true, the database waits before responding.

Example in MySQL:

```sql
1' AND SLEEP(5) -- 
```

If the page loads after 5 seconds, the condition is true.

### Out-of-Band SQL Injection

In some cases, the attacker cannot see output directly and cannot use blind methods easily. Then the database is forced to send data to another system controlled by the attacker, such as DNS or HTTP requests.

For example, sensitive data may be sent to an external server through DNS queries.

This type is less common but useful in restricted environments.
